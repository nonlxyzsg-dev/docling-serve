## 1. Что это

Форк [docling-serve](https://github.com/docling-project/docling-serve) — FastAPI-сервиса, оборачивающего библиотеку `docling` в HTTP API. Принимает файлы через `/v1/convert/file`, отдаёт markdown / JSON. Внутри использует `docling-jobkit` как orchestrator (Local mode у нас).

**Это главный entry point форка.** Здесь же лежат `TASKS/` и `ANALYSIS_REPORT.md` — координация задач, которые могут затрагивать `docling-serve`, `docling`, или оба сразу.

## 4. Главная боль

При обработке документов с большим количеством картинок реальный параллелизм запросов к VLM-бэкенду намного ниже теоретически возможного.

**Конкретный пример:** документ 207 страниц / 84 картинки.
- SGLang (наш VLM backend) имеет 28 параллельных слотов, очередь пустая всё время
- Расчётное время: 84 картинки × ~10 сек / параллелизм 20 ≈ 1.5–2 минуты
- Реальное время: **13 минут**
- Метрики SGLang показывают «волны»: 60–100 секунд активной работы при потоке ~14–16 запросов, затем 30–60 секунд простоя (поток падает до 0–1), затем снова волна. Из 13 минут ~7 минут реальная работа, ~6 минут простой.

Подозрение: docling-serve / docling формирует батчи с синхронным барьером — все запросы в батче должны завершиться, прежде чем стартует следующий. Параметр `vlm_concurrency` уважается **внутри батча**, но не убирает барьеры **между батчами**.

Цель форка — устранить эти барьеры.

## 5. Долгосрочное направление

В перспективе хотим, чтобы docling-serve мог использовать наш **OCR SDK** (отдельный сервис на Qwen3.5-2B + PP-DocLayoutV3 с параллелизмом 32 воркера) как один из recognition backend'ов вместо только OpenAI-compatible VLM API.

OCR SDK обрабатывает аналогичный документ за 1–1.5 минуты вместо 13 минут docling. Это эталон по скорости для нас.

Сейчас не реализовываем, но в анализе нужно учесть: где в коде сейчас захардкожено «recognition = OpenAI-compatible VLM call», и какая абстракция позволит подключить новый backend без переписывания pipeline.

## 6. Принципы работы

### 6.1. Минимально-инвазивные изменения
Каждый патч — будущий конфликт при rebase на upstream. Поэтому:
- Не переписывай файлы целиком, если можно точечно
- Группируй наши изменения в выделенные модули, а не размазывай по кодовой базе
- Каждый патч — отдельный коммит с префиксом `[fork]` в title
- Веди `CHANGELOG_FORK.md` в корне репозитория

### 6.2. Стратегия синхронизации с upstream
- Главная ветка: `main` (наша рабочая)
- Ветка `upstream-tracking` — еженедельный merge из upstream/main
- При rebase каждый `[fork]`-коммит пересматривается. Если соответствующий issue закрыт в upstream — наш патч удаляется.

### 6.3. Тесты обязательны
Любое изменение производительности — с бенчмарком на эталонном наборе документов (`benchmarks/`). Никаких «кажется быстрее».

### 6.4. Совместимость
- HTTP API наружу — не меняем
- Имена параметров в JSON — не меняем
- Поведение для документов, которые сейчас работают приемлемо — не ломаем

### 6.5. Когда сомневаешься — спрашивай
Это код в продакшене. Если решение неочевидно — формулируй вопрос и жди ответа. Не угадывай.

### 6.6. Безопасность
- Никогда не вшивай реальные API-ключи / токены / пароли
- В примерах — плейсхолдеры (`<API_KEY>`)

## 7. Технический контекст инфраструктуры

### 7.1. Что вокруг docling-serve
- **Прокси перед docling-serve:** `docling-proxy` (FastAPI), порт 5005. Принимает запросы от OpenWebUI, маршрутизирует, инжектирует VLM-конфиг. Карта известных багов upstream — там в коде и в `docling_proxy_passport_v3.md` (раздел 12).
- **Библиотека docling:** наш форк https://github.com/nonlxyzsg-dev/docling. Устанавливается в Docker-образ как pip-зависимость.
- **VLM backend:** SGLang v0.5.9 с моделью Qwen3.5-122B-agent-50k, OpenAI-compatible API через LiteLLM (порт 4000). 28 параллельных слотов, контекст 50K, без reasoning. На GPU H200 #0 сервера tvr-srv-aiprod (10.121.3.190).
- **OCR SDK (отдельный сервис):** Qwen3.5-2B vLLM + PP-DocLayoutV3, 32 воркера, ~5 страниц/сек. На сервере tvr-srv-ai (10.121.3.201:9996). Сейчас не используется docling-serve, в перспективе — будет.
- **Vector store:** Milvus, эмбеддинги BGE-m3
- **Frontend:** OpenWebUI

### 7.2. Production deployment
- **Сервер:** tvr-srv-ai (10.121.3.201)
- **Образ сейчас:** `ghcr.io/docling-project/docling-serve:main` (CPU only, без GPU)
- **Конфигурация:** `UVICORN_WORKERS=2`, `DOCLING_SERVE_ENG_LOC_NUM_WORKERS=2`
- **PaddleOCR отключён** на уровне нашего прокси (`do_ocr=false` инжектируется), всё распознавание идёт через VLM
- **Compose:** `/docker-docling/docker-compose.yml`
- **Healthcheck:** реальная конверсия тестового файла + label `autoheal=true`

### 7.3. Типичная нагрузка
- PDF (тексты, сканы, смешанные), DOCX (включая legacy с OLE-формулами от MathType), XLSX, PPTX, HTML, .DOC из Confluence (MIME HTML)
- Размеры: от 1 до 300+ страниц
- Картинки: 0–200 на документ
- Языки: преимущественно русский
- Параллельная нагрузка: типично 1–3 документа одновременно, пиково до 10

### 7.4. Известные баги, которые сейчас обходим снаружи (в прокси или sed-патчем в Dockerfile)
1. **`docling_jobkit/orchestrators/local/worker.py:126`** — `SyntaxError: keyword argument repeated: exc_info`. Патчим sed'ом в Dockerfile.
2. **Pillow `tile cannot extend outside image`** при `image_export_mode=embedded` + VLM pipeline. Прокси подменяет на `placeholder`.
3. **`do_picture_description=true` при VLM pipeline** инициализирует SmolVLM (`FileNotFoundError`). Прокси подавляет.
4. **`picture_description_custom_config` не уважает `concurrency`** — последовательная обработка картинок. Прокси переключает на `picture_description_api` (×2.2 ускорение, но барьеры между батчами остаются — главная боль).

В рамках форка нужно решить: фиксить ли нативно (тогда упрощаем прокси) или оставить как есть.
