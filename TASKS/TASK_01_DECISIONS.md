## ВАЖНОЕ АРХИТЕКТУРНОЕ УТОЧНЕНИЕ от инженера

Прежде чем перейти к новому плану — концептуальное уточнение по подходу.

**Наша инфраструктура:**
- VLM backend = SGLang (28 параллельных слотов, prefix cache, очередь)
- Recognition backend (в перспективе) = OCR SDK (32 воркера, очередь)
- Оба backend'а **сами управляют batching и scheduling**

**Из этого следует:** искусственное чанкование в docling (`elements_batch_size=16`, барьеры между батчами) — это **архитектурный анахронизм**, унаследованный от случаев, когда docling работает с маленькими локальными моделями или OpenAI API с rate limits. Для нашей инфраструктуры это **активно вредит**:
- Теряются оптимизации SGLang prefix cache
- Создаются простои при дрейне батчей
- Расщепляется параллелизм, который backend и так умеет делать сам

**Желаемое поведение:**
1. Извлечь структуру документа за секунды (CPU)
2. Параллельно (overlap) — отправить ВСЕ найденные картинки в backend как только они появляются, без чанкования
3. Backend сам разруливает очередь (28 параллельных, остальные ждут — это его работа, не наша)
4. Получать ответы по мере готовности (через id, не через позицию в списке)
5. Вставлять в нужное место документа сразу при получении

**Для этого нужны механизмы:**
- **Глобальный semaphore** «максимум N запросов в полёте на документ» (не размер батча!) — защита от перегрузки backend, дефолт большой (например, 64)
- **Streaming submit** — отправлять картинки по мере обнаружения, не дожидаясь окончания layout-фазы
- **Result fan-in by ID** — каждый запрос помечен `picture_id`, ответы матчатся через map, не через порядок
- **Параметр `no_batching: bool`** в опциях — если true, элементы шлются по одному без чанкования (просто пачка submit'ов с семафорной защитой)

Это **не оптимизация существующей логики**, это **смена парадигмы**. Поэтому TASK_03 объединяется с TASK_07 в одну архитектурную задачу (см. ниже).

### TASK_03 — Архитектурный сдвиг: убрать batching, ввести абстракцию backend, overlap фаз

**Сложность:** L (объединены прежние TASK_03 + TASK_07 + overlap фаз)

**Что делаем:**

#### Часть A: ввод абстракции `RecognitionBackend`

- Создать интерфейс `RecognitionBackend` (Protocol) с методами:
  - `describe_picture(picture_id, image, prompt, **kwargs) -> RecognitionResult` — async или sync, обсуждаемо
  - Опционально `describe_batch(...)` — для backend'ов, где batch имеет смысл
- Реализовать `OpenAiCompatibleBackend(RecognitionBackend)` — обёртка над текущим `api_image_request`
- Подготовить место для `OcrSdkBackend(RecognitionBackend)` — **не реализовывать**, оставить заглушку с `NotImplementedError` и комментарием. Реализация — TASK_08
- Конфигурация через discriminated union в `pipeline_options.py` (kind="openai_compatible" | "ocr_sdk")

#### Часть B: переписать picture description под "submit all → fan-in"

- `PictureDescriptionApiModel._annotate_images` переписать на новую модель:
  - Не создавать `ThreadPoolExecutor` per batch
  - Использовать **долгоживущий пул задач + asyncio.Semaphore** (или ThreadPoolExecutor + Semaphore)
  - Submit всех картинок документа сразу (или по мере поступления — см. часть C)
  - Fan-in результатов через `picture_id`, не через порядок
- Добавить параметр в `PictureDescriptionApiOptions`:
  - `no_batching: bool = False` (или эквивалент по смыслу — назови как считаешь правильным)
  - `max_concurrent_in_flight: int = 64` — глобальный лимит, не размер батча
- Обеспечить обратную совместимость: если `no_batching=False` (дефолт) — старое поведение; если `True` — новое

#### Часть C: overlap фаз (text extraction + recognition)

- Сейчас `_build_document` (pages) полностью завершается перед `_enrich_document` (pictures). Это последовательно
- Изменить так, чтобы как только в page обнаружена картинка, она сразу отправлялась на recognition (через `RecognitionBackend.describe_picture`)
- Реализация: producer-consumer между фазами. Page processing — producer картинок; recognition — consumer
- Текст и layout доделываются параллельно с recognition
- В конце — barrier на «все картинки обработаны», только потом возврат результата

#### Часть D: отключить `picture_classification` по умолчанию

- VLM в описании сама пишет тип картинки («это фото», «это диаграмма»)
- Текущая classification — CPU, последовательная, медленная, дублирует работу VLM
- Решение: добавить параметр в опции pipeline `do_picture_classification: bool = False` (поменять дефолт с True на False) либо выводить классификацию из стандартного pipeline'а
- При желании пользователь может включить обратно

**Тесты обязательны:**
- Бенчмарк до/после на всех 5 эталонных документах
- Регрессионный тест: `md_content_bytes` не должен измениться более чем на ±5% относительно baseline
- Корректность: визуально сравнить markdown на 1-2 документах

**Ожидаемый эффект:**
- На `Поражай.pdf` (207/84): с 13 мин → **ориентир ≤ 3 мин**
- На простых документах: без регрессии (±5%)
- На сканах через VLM pipeline: без изменений (TASK_03 не трогает VLM pipeline, только picture_description)

**Риски:**
- Высокий риск merge-конфликтов при rebase — `base_pipeline.py` активно меняется в upstream. Делать минимально-инвазивно.
- Возможна проблема с памятью если submit ALL для документа с 500 картинками — продумать backpressure через semaphore
- Overlap фаз может вскрыть скрытые предположения о порядке в коде — внимательно смотреть

**Когда заканчивается:** отчёт `TASKS/TASK_03_REPORT.md` с benchmark-результатами до/после.
