## Контекст

Перед началом — **обязательно** прочитай `CLAUDE.md` в этом репозитории. Затем — `CLAUDE.md` в репозитории `docling` (https://github.com/nonlxyzsg-dev/docling). Там общий контекст и принципы. Этот файл — конкретная задача в рамках этого контекста.

Кратко: мы форкаем `docling` и `docling-serve`. Главная боль — низкий реальный параллелизм при обработке документов с большим количеством картинок. На документе 207 страниц / 84 картинки docling-serve работает 13 минут вместо ожидаемых 1.5–2 минут.

Метрики SGLang во время обработки показывают «волны»: 60–100 секунд работы при потоке ~14–16 параллельных запросов, потом 30–60 секунд простоя (поток падает до 0–1), потом снова волна. SGLang при этом свободен — очередь пустая, есть свободные слоты.

Для сравнения: наш отдельный OCR SDK (Qwen3.5-2B + PP-DocLayoutV3, 32 воркера) обрабатывает аналогичный документ за 1–1.5 минуты. Это эталон, к которому стремимся.

### Шаг 2 — Разобраться с источником «барьеров» между батчами

**Это центральный вопрос анализа.** Метрики SGLang показывают паттерн:
```
T+0:    запросов 0
T+10:   запросов 15  ← волна началась
T+90:   запросов 14
T+100:  запросов 0   ← волна закончилась, ПРОСТОЙ
T+150:  запросов 0   ← продолжается простой
T+170:  запросов 16  ← следующая волна
```

Найти в коде, **где формируются батчи** для VLM-запросов и **почему между батчами есть синхронный барьер** (видимо, ждётся завершение всех запросов батча перед началом следующего).

Ключевые места для проверки (в обоих репозиториях — `docling` и `docling-serve`):
- `picture_description_api` — где создаются батчи запросов на описание картинок. Параметр `concurrency` — как используется (semaphore? ThreadPoolExecutor? asyncio.gather?).
- `vlm_pipeline` — где формируется batch страниц для full-page OCR.
- `page_batch_size` — упомянут в Issue #419 как захардкоженный 4. Найти где, можно ли вынести в конфиг.
- `batch_size` для `picture_classification` (отдельная проблема — работает на CPU, последовательно).

Изучить open issues:
- https://github.com/docling-project/docling-serve/issues/419 (page_batch_size, ~2x вместо ~10x)
- https://github.com/docling-project/docling-serve/issues/425 (parallel processing in RQ mode)
- https://github.com/docling-project/docling-serve/issues/463 (env vars for VLM defaults)
- https://github.com/docling-project/docling-serve/issues/318 (VLM API не вызывается)
- https://github.com/docling-project/docling-serve/issues/257 (общая производительность)
- https://github.com/docling-project/docling/discussions/2285 (thread-safety)
- https://github.com/docling-project/docling/issues/2635 (зависание VLM pipeline)

### Шаг 3 — Найти и каталогизировать все известные баги, которые мы патчим снаружи

Из `CLAUDE.md` (раздел 7.4 в docling-serve, 6.3 в docling) — четыре известных бага. Найти их в коде, оценить:
- Можно ли пофиксить нативно в форке (минимально-инвазивно)

### Шаг 4 — Оценить thread-safety

Issue #2285 говорит о проблемах с `DocumentConverter` и `HybridChunker`. У нас в проде: `UVICORN_WORKERS=2`, `DOCLING_SERVE_ENG_LOC_NUM_WORKERS=2`. Если thread-safety реально нарушена — это потенциальный источник плавающих багов.

### Шаг 5 — Архитектурная подготовка к сменному backend

В `CLAUDE.md` (раздел 5) сказано: в перспективе хотим использовать наш `ocr-sdk` как один из backend'ов вместо только OpenAI-compatible VLM API. Сейчас не реализовываем, но в анализе:
- Указать, **где в коде сейчас захардкожено** предположение «recognition = OpenAI-compatible VLM call»
- Предложить минимальную абстракцию (интерфейс / протокол), которая сделает смену backend возможной без переписывания pipeline
- Эта абстракция не входит в текущую задачу, но в плане работ должна быть отдельным этапом
