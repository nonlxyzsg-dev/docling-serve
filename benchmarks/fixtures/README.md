# fixtures — эталонные документы

> Сюда кладутся реальные (анонимизированные) документы для benchmark-прогонов.

## Зачем структура по подкаталогам

Документы разбиты не по размеру, а по **типу**, потому что разные типы проверяют разные части pipeline'а:

| Подкаталог              | Pipeline / Что проверяем                                                                                                                                    |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `pdf_text_large/`       | **главный кейс.** VLM pipeline с большим количеством картинок — проверяем устранение барьеров параллелизма.                                                 |
| `pdf_formulas/`         | VLM pipeline на страницах с формулами — проверяем, что формулы распознаются и barrier-рефакторинг не ломает результат.                                      |
| `pdf_text_small/`       | Sanity check: маленький документ должен обрабатываться за секунды. Если на нём регрессия — значит, мы добавили лишние накладные расходы в fork-патчах.     |
| `pdf_scan_cyrillic/`    | VLM full-page OCR на скане (кириллица). Проверяет совместимость барьерного рефакторинга с фулл-пэйдж рисунками.                                             |
| `docx_with_ole/`        | DOCX-парсер + embedded formulas. Не VLM, но должен работать — проверяем, что мы ничего не ломаем в не-VLM пути.                                             |

## Правила

1. **Анонимизация:** все PII / коммерческая тайна должны быть замазаны или убраны до коммита. Если анонимизировать невозможно — держать файл вне git и положить `.external` placeholder с описанием, где файл живёт на сервере.
2. **Размер:** если суммарный размер fixtures превышает ~20 МБ, перейти на `git lfs` или внешнее хранилище (см. открытый вопрос в `TASKS/TASK_02_infrastructure.md`).
3. **Имена файлов:** оригинальные имена сохраняем (они упомянуты в `CLAUDE.md` и `TASK_01_DECISIONS.md`), чтобы потом не искать, что с чем соотносится.
4. **Новые документы:** каждый новый добавленный документ — строка в этом README с характеристиками и причиной добавления.

## Placeholder-файлы

До заливки реальных документов в каталогах лежат `.gitkeep` — пустые файлы, чтобы структура попала в git. При заливке реальных документов их можно удалить.
